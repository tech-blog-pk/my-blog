JavaScript Promises: A Complete Beginner's Guide

MK
Malik Khan · May 3, 2026 9 min read

If you've written any JavaScript beyond the basics, you've run into Promises. Maybe you've used them without fully understanding what's happening under the hood. That's what this guide is for — a clear, practical explanation of how Promises work, and how to use them well.

What Problem Do Promises Solve?

JavaScript is single-threaded, which means it can only do one thing at a time. But many operations — fetching data from an API, reading a file, querying a database — take time. If JavaScript just waited (blocked) for each of these to finish, your app would freeze.

The old solution was callbacks: pass a function that gets called when the work is done. This worked, but it quickly became unreadable when you had multiple async operations depending on each other — a problem so common it earned the name "callback hell."

Promises were introduced in ES2015 to fix this. A Promise represents a value that isn't available yet but will be resolved (or rejected) in the future.

The Three States of a Promise

Every Promise is in one of three states at any given time:

  • Pending: The operation hasn't completed yet.
  • Fulfilled: The operation succeeded and the Promise has a value.
  • Rejected: The operation failed and the Promise has a reason (error).

Once a Promise moves from pending to either fulfilled or rejected, it's settled — and it stays that way. It can't go back to pending, and it can't switch from fulfilled to rejected.

Creating a Promise

You create a Promise by passing a function (the executor) to the Promise constructor. That function receives two arguments: resolve and reject.

const myPromise = new Promise((resolve, reject) => {
  // simulate async work
  setTimeout(() => {
    const success = true;

    if (success) {
      resolve('Data loaded successfully');
    } else {
      reject(new Error('Something went wrong'));
    }
  }, 1000);
});

Call resolve(value) when the work succeeds. Call reject(error) when it fails. Simple.

Consuming Promises with .then() and .catch()

To use the value from a Promise, you chain .then(). To handle errors, you use .catch().

myPromise
  .then((result) => {
    console.log(result); // 'Data loaded successfully'
  })
  .catch((error) => {
    console.error(error.message); // 'Something went wrong'
  });

The key insight: .then() also returns a Promise, which means you can chain them.

Chaining Promises

This is where Promises shine over callbacks. Instead of nesting functions inside functions, you chain .then() calls in a flat, readable sequence.

fetchUser(userId)
  .then((user) => fetchUserPosts(user.id))
  .then((posts) => filterPublished(posts))
  .then((published) => {
    console.log('Published posts:', published);
  })
  .catch((error) => {
    // any error from any step lands here
    console.error('Failed:', error.message);
  });
Important: A single .catch() at the end of a chain will catch errors from any step above it. This makes error handling much cleaner than the callback approach.

Promise.all() — Run Multiple Promises in Parallel

If you have multiple independent async operations, don't run them one after the other — run them at the same time.

const [users, posts, comments] = await Promise.all([
  fetchUsers(),
  fetchPosts(),
  fetchComments()
]);

// all three finish before this line runs

Promise.all() resolves when all the Promises resolve. If any one of them rejects, the whole thing rejects immediately. Use it when you need all results before continuing.

Promise.allSettled() — When You Want All Results Regardless

Sometimes you want to run multiple Promises and get all results — even if some fail. That's what Promise.allSettled() is for.

const results = await Promise.allSettled([
  fetchUsers(),
  fetchPosts(),
  fetchComments()
]);

results.forEach((result) => {
  if (result.status === 'fulfilled') {
    console.log('Success:', result.value);
  } else {
    console.error('Failed:', result.reason);
  }
});

The Relationship Between Promises and Async/Await

Async/await is not a different system — it's syntax sugar on top of Promises. Every async function returns a Promise. Every await expression unwraps a Promise.

// These two are equivalent:

// Promise style
fetchUser(id)
  .then(user => console.log(user))
  .catch(err => console.error(err));

// Async/await style
async function loadUser(id) {
  try {
    const user = await fetchUser(id);
    console.log(user);
  } catch (err) {
    console.error(err);
  }
}

Async/await is usually easier to read — especially for sequential operations. But understanding the underlying Promise mechanics helps you debug issues and write better code.

Common Mistakes to Avoid

1. Forgetting to return in .then()

If you forget to return inside a .then(), the next step in the chain gets undefined instead of your value.

// WRONG — next .then() gets undefined
fetchUser(id)
  .then((user) => {
    fetchUserPosts(user.id); // forgot return!
  })
  .then((posts) => console.log(posts)); // undefined

// CORRECT
fetchUser(id)
  .then((user) => {
    return fetchUserPosts(user.id);
  })
  .then((posts) => console.log(posts));

2. Not handling errors

An unhandled Promise rejection will crash Node.js and silently swallow errors in the browser. Always add a .catch() — or wrap your await calls in try/catch.

Summary

Promises give JavaScript a clean way to handle asynchronous operations. Key things to remember:

  • A Promise is either pending, fulfilled, or rejected
  • Use .then() for success and .catch() for errors
  • Chain .then() calls instead of nesting callbacks
  • Use Promise.all() to run independent operations in parallel
  • Async/await is built on top of Promises — learn both

Once Promises click, the rest of async JavaScript becomes much easier to reason about. Happy coding.
