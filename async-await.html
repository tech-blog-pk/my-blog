Mastering Async/Await: Patterns You're Probably Missing

Most developers only know the basics of async/await. The real performance gains come from parallel execution, smart error handling, and patterns that rarely get covered in tutorials.

AK
Ali Khan · Apr 22, 2026 · 8 min read
JavaScriptAsyncPerformance

Async/await transformed the way we write JavaScript. It replaced callback hell and made Promise chains readable. But most developers stop at the basics — and that means they're leaving serious performance gains on the table.

In this article, we'll cover five patterns that go beyond the basics and help you write faster, cleaner, and more resilient async JavaScript.

1. What Is Async/Await, Really?

Async/await is syntactic sugar over Promises. When you use await, JavaScript pauses that function until the Promise resolves — but it does not block the entire program.

JavaScript
// An async function always returns a Promise
async function getData() {
  const response = await fetch('https://api.example.com/data');
  const data = await response.json();
  return data;
}

Key Rule: Every async function returns a Promise — even if you return a plain value. JavaScript wraps it automatically.

2. Parallel Execution with Promise.all()

This is the most impactful pattern most developers miss. If you have multiple async operations that don't depend on each other, run them in parallel — not one after another.

❌ Wrong Way — Sequential (Slow)

JavaScript — Avoid this
async function loadDashboard() {
  // Each awaits the previous — 3 seconds total
  const user  = await getUser();   // 1 second
  const posts = await getPosts(); // 1 second
  const stats = await getStats(); // 1 second
  return { user, posts, stats };
}

✅ Right Way — Parallel (3x Faster)

JavaScript — Do this
async function loadDashboard() {
  // All three run at the same time — ~1 second total
  const [user, posts, stats] = await Promise.all([
    getUser(),
    getPosts(),
    getStats()
  ]);
  return { user, posts, stats };
}

By using Promise.all(), all three requests fire simultaneously. This can make your app feel dramatically faster, especially on slower connections.

3. Error Handling Done Right

Unhandled async errors are one of the most common bugs in JavaScript apps. Always wrap your async operations in try/catch — and don't forget to check HTTP status codes.

JavaScript
async function fetchUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`);

    // fetch() doesn't throw on 404 — you must check
    if (!response.ok) {
      throw new Error(`Server error: ${response.status}`);
    }
    return await response.json();

  } catch (error) {
    console.error('Failed to fetch user:', error.message);
    return null;
  }
}

Common mistake: fetch() only throws on network failures. A 404 or 500 response does NOT throw — always check response.ok manually.

4. Promise.allSettled() — When Partial Failure Is Okay

Promise.all() fails fast — if one Promise rejects, everything fails. Sometimes you want results from what succeeded, even if some failed. That's Promise.allSettled().

JavaScript
const results = await Promise.allSettled([
  fetchUser(),
  fetchPosts(),
  fetchComments()
]);

results.forEach((result, index) => {
  if (result.status === 'fulfilled') {
    console.log(`Request ${index} succeeded:`, result.value);
  } else {
    console.warn(`Request ${index} failed:`, result.reason);
  }
});

5. Async Loops — The Right Approach

Mixing async/await with .forEach() is a common trap. Use for...of for sequential execution, or Promise.all with .map() for parallel.

JavaScript
const userIds = [1, 2, 3, 4, 5];

// ✅ Parallel — fastest
const users = await Promise.all(
  userIds.map(id => fetchUser(id))
);

// ✅ Sequential — when order matters
for (const id of userIds) {
  const user = await fetchUser(id);
  processUser(user);
}

// ❌ forEach ignores the Promise — don't use this
userIds.forEach(async (id) => {
  const user = await fetchUser(id);
});

Summary

Five patterns to remember:

✅ Promise.all() — run independent operations in parallel
✅ Promise.allSettled() — collect all results even if some fail
✅ Try/catch — always handle errors in async functions
✅ Check response.ok — fetch() doesn't throw on HTTP errors
✅ for...of over forEach — for sequential async loops

Apply these patterns and your JavaScript will be faster, safer, and much easier to debug. Happy coding!
